# 开发笔记

## 项目架构决策

### 为什么选 FastAPI + Vue3?
- FastAPI: 异步高性能，自动生成API文档 (Swagger)，Python生态MIB库丰富
- Vue3 + Element Plus: 组件库成熟，表单/表格/上传等组件开箱即用
- 前后端分离: 开发独立，前端热更新，后端独立重启

### 为什么用 pysmi + pysnmp?
- pysmi: MIB文件编译器，将 .mib 文本格式编译为Python可读的结构
- pysnmp: SNMP协议库，提供MIB树遍历、OID查询能力
- 两者配合: pysmi负责解析MIB语法，pysnmp负责构建OID树

### 为什么用 lxml 生成 XML?
- 标准库 xml.etree 功能够用但格式化能力弱
- lxml 支持 `etree.indent()` 自动缩进
- lxml 的 XMLSyntaxError 提供精确的语法校验位置

## 踩坑记录

### 坑1: pysmi编译MIB时找不到依赖
**现象**: 编译自定义MIB时报 "Missing MIB module: SNMPv2-MIB"
**原因**: pysmi默认搜索路径不包含系统MIB库
**解决**: 配置多个MIB搜索路径 (UPLOAD_DIR + DATA_DIR/mibs + ~/.pysnmp/mibs)

### 坑2: UUID重复导致Zabbix导入失败
**现象**: Zabbix导入时报 "Details: UUID already exists"
**原因**: 随机UUID碰撞 (概率极低但确实发生)
**解决**: 生成时检查 `_used_uuids` 集合，确保全局唯一

### 坑3: XML中特殊字符未转义
**现象**: 触发器expression中的 `<>` 符号破坏XML结构
**原因**: lxml默认不转义文本节点中的特殊字符
**解决**: 使用 `&lt;&gt;` 手动转义，或在生成时用 `etree.CDATA` 包裹

### 坑4: Zabbix导入时 groups 顺序不对
**现象**: 导入后模板不显示在预期的主机组中
**原因**: XML中groups的书写顺序必须与Zabbix数据库已有组匹配
**解决**: 固定两组 "服务器模板" 和 "监控模板"，不支持自定义

### 坑5: key中包含中文导致Zabbix报错
**现象**: Zabbix提示 "Invalid key format"
**原因**: 自动生成的key可能包含翻译后的中文
**解决**: `generate_zabbix_key()` 函数强制只保留 `a-z0-9.` 字符

### 坑6: TABLE类OID和SCALAR OID的区分
**现象**: 有些OID看起来像独立项但实际是表格成员
**原因**: MIB中TABLE的INDEX定义方式多样 (INDEX/AUGMENTS/IMPLIED)
**解决**: 检查sym的 `index` 和 `augments` 属性，有值则标记为indexed

## 技术方案对比

### XML生成方案
| 方案 | 优点 | 缺点 |
|------|------|------|
| 字符串拼接 | 简单直接 | 易出错，缩进难维护 |
| xml.etree | 标准库，无需安装 | 格式化能力弱 |
| **lxml** | 格式化好，校验强 | 需要额外安装 |
| Jinja2模板 | 模板可读性好 | 复杂逻辑难嵌入 |

**选择 lxml**: 格式化+校验是刚需，且lxml是Python XML处理的事实标准。

### 数据存储方案
| 方案 | 优点 | 缺点 |
|------|------|------|
| **SQLite** | 轻量无需安装，单文件 | 不适合高并发 |
| PostgreSQL | 功能强大 | 部署复杂 |
| JSON文件 | 极简 | 不支持查询 |

**选择 SQLite**: 本工具是单用户场景，SQLite完全够用，且无需额外安装数据库服务。

### 前端状态管理
| 方案 | 优点 | 缺点 |
|------|------|------|
| **Pinia** | Vue3官方推荐，简洁 | 需要学习 |
| Vuex | 生态成熟 | 略重 |
| 纯props/emit | 无需学习 | 组件多了难维护 |

**选择 Pinia**: Vue3官方状态管理，API简洁。

## Zabbix 6.4 XML 格式备忘

### 文档结构
```
zabbix_export
├── version (固定6.4)
├── date (UTC时间)
├── groups[]
│   └── group {uuid, name}
├── templates[]
│   └── template {uuid, template, name, groups[], items[], discovery_rules[], macros[]}
```

### item 字段顺序
```
uuid → name → type → snmp_oid → key → delay → history → trends → value_type → units → tags
```

### discovery_rule 字段顺序
```
uuid → name → type → snmp_oid → key → delay → lifetime → item_prototypes[]
```

### item_prototype 字段顺序
```
uuid → name → type → snmp_oid → key → delay → history → trends → value_type → units → tags → trigger_prototypes[]
```

### trigger_prototype 字段顺序
```
uuid → expression → name → priority → manual_close
```

### value_type 取值
- 0 = FLOAT (浮点数)
- 1 = CHARACTER (字符)
- 2 = LOG (日志)
- 3 = UNSIGNED (无符号整数)
- 4 = TEXT (文本)

### delay 合法值
- 1m, 5m, 10m, 1d

### type 取值
- SNMP_AGENT (本项目固定使用此值)

## Python 3.12 兼容性踩坑 (实测记录)

### 坑7: pysmi 0.3.4 不兼容 Python 3.12
**现象**: `ModuleNotFoundError: No module named 'imp'`
**原因**: Python 3.12 移除了 `imp` 模块，pysmi 0.3.4 依赖它
**解决**: 升级到 pysmi 1.5.x，API 从 `MibCompiler(writer, searcher, reader)` 变为 `MibCompiler(parser, codegen, writer)` + `add_sources()`/`add_searchers()`

### 坑8: pysnmp 4.4.12 不兼容 Python 3.12
**现象**: `ModuleNotFoundError: No module named 'asyncore'`
**原因**: Python 3.12 移除了 `asyncore` 模块
**解决**: 安装 `pyasyncore` 和 `pyasynchat` 回退包

### 坑9: pyasn1 0.6.x 与 pysnmp 4.x 不兼容
**现象**: `ModuleNotFoundError: No module named 'pyasn1.compat.octets'`
**原因**: pyasn1 0.6.x 移除了 compat 模块
**解决**: 固定 pyasn1==0.4.8

### 坑10: requirements.txt 中文注释导致 pip 解码失败
**现象**: `UnicodeDecodeError: 'gbk' codec can't decode`
**原因**: Windows 下 pip 默认用 GBK 读取 requirements.txt，中文字符解码失败
**解决**: requirements.txt 中避免使用中文字符

### pysmi 1.5.x API 变化
```python
# 旧 API (0.3.x):
compiler = MibCompiler(writer, searcher, reader)
results = compiler.compile('MIB_NAME')

# 新 API (1.5.x):

compiler.add_sources(reader)
compiler.add_searchers(searcher)
results = compiler.compile('MIB_NAME')
```

## MIB解析性能优化 (实测记录)

### 问题
Lenovo MIB文件 (163KB, 257个OID) 解析耗时 ~5300ms，用户体感明显卡顿。

### 瓶颈定位
用 `time.time()` 分段计时：
| 阶段 | 耗时 | 占比 |
|------|------|------|
| 文件读取 | 1ms | 0% |
| 正则解析MIB | 5312ms | 99.6% |
| 自动分类 | 17ms | 0.3% |
| 翻译 | 5ms | 0.1% |

**瓶颈在正则解析**，具体是 `_find_oid_definition()` 方法。

### 根因分析
原始实现对每个OID都要：
1. 调用 `_resolve_parent_oid()` → `_find_oid_definition()`
2. `_find_oid_definition()` 对整个MIB内容做 4~5 次正则搜索 (`re.DOTALL`)
3. 递归调用（parent的parent也要查找）

257个OID × 4次正则 × 递归深度 = **~1000次全文正则扫描**

### 优化方案: 预构建查找表 + 批量解析

**核心思路**: 一次性扫描MIB，构建 `name → (parent, subid)` 字典，再用缓存递归解析所有OID。

```python
# 优化前: 每个OID单独正则搜索全文
for oid in oids:
    oid_str = resolve_parent_oid(parent, subid, content)  # 每次4~5个正则

# 优化后: 一次性扫描 + 缓存
oid_defs = {}  # 预构建的 name→(parent,subid) 字典
resolved_cache = {}  # 已解析的 name→完整OID 缓存

# 第一遍: 一个正则提取所有定义
for m in def_pattern.finditer(content):
    oid_defs[name] = (parent, subid)

# 第二遍: 带缓存的递归解析
def resolve(name):
    if name in resolved_cache: return resolved_cache[name]
    parent, subid = oid_defs[name]
    result = resolve(parent) + "." + subid
    resolved_cache[name] = result
    return result
```

### 其他优化
1. **AUGMENTS正则短路**: 先检查 `"AUGMENTS" in content` 再跑正则，避免654ms空扫描
2. **表格检测复用查找表**: `_detect_table_indices_fast()` 直接查 `resolved_cache`，不再调用 `_find_oid_definition()`

### 优化结果
| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 解析耗时 | 5300ms | 27ms | **197x** |
| API端到端 | ~5500ms | 137ms | **40x** |
| OID数量 | 257 | 252 | (过滤了重复) |
| Indexed | 190 | 190 | 不变 |

### 关键代码位置
- 优化后的解析器: `src/backend/app/services/mib_parser.py` → `_parse_mib_text()`
- 预构建查找表: `def_pattern` + `oid_defs` 字典
- 缓存递归: `resolve()` 内部函数 + `resolved_cache` 字典
- 快速表格检测: `_detect_table_indices_fast()`

## Docker 构建失败: 内网服务器无法下载依赖

### 现象
在 Rocky Linux 8.10 内网服务器上执行 `docker-compose build`，后端镜像构建失败：
```
apt-get update → Unable to connect to deb.debian.org:http
E: Unable to locate package gcc
E: Unable to locate package libxml2-dev
```

### 原因
`python:3.12-slim` 基础镜像是 Debian 系统。构建时 `apt-get update` 需要访问 `deb.debian.org` 下载包索引，但内网服务器没有外网访问权限（或 DNS 不通），导致：
1. `apt-get update` 无法连接 Debian 源
2. 后续 `apt-get install gcc libxml2-dev libxslt1-dev` 找不到包

### 解决方案

#### 方案一: 配置 Docker 使用国内镜像源 (推荐)
在 Dockerfile 中把 Debian 源换成国内镜像：
```dockerfile
RUN sed -i 's/deb.debian.org/mirrors.aliyun.com/g' /etc/apt/sources.list.d/debian.sources
```
前提：服务器能访问 `mirrors.aliyun.com`

#### 方案二: 配置 Docker 使用宿主机 DNS
创建/编辑 `/etc/docker/daemon.json`：
```json
{
  "dns": ["你的内网DNS地址", "114.114.114.114"]
}
```
然后 `systemctl restart docker`

#### 方案三: 预构建镜像离线导入 (纯内网推荐)
1. 在有外网的机器上构建镜像并保存：
   ```bash
   docker build -t mib-backend ./backend
   docker save mib-backend > mib-backend.tar
   ```
2. 把 `.tar` 文件传到内网服务器
3. 在内网服务器上加载：
   ```bash
   docker load < mib-backend.tar
   ```
4. 修改 `docker-compose.yml`，去掉 `build` 改用 `image: mib-backend`

#### 方案四: 去掉 lxml 编译依赖 (最简单)
lxml 有预编译的 wheel 包，不需要从源码编译。把 Dockerfile 中的 `apt-get` 整段删掉：
```dockerfile
# 删除这段:
# RUN apt-get update && apt-get install -y --no-install-recommends
#     gcc libxml2-dev libxslt1-dev
#     && rm -rf /var/lib/apt/lists/*
```
pip 安装 lxml 时会自动下载预编译 wheel，不需要 gcc。
**但前提是 pip 源也能访问**，否则还需要配置 pip 国内镜像。

#### 方案五: pip 也换国内源
在 Dockerfile 中：
```dockerfile
RUN pip install -i https://pypi.tuna.tsinghua.edu.cn/simple --trusted-host pypi.tuna.tsinghua.edu.cn ...
```

### 推荐做法
对于纯内网服务器，**方案三（离线导入）最稳妥**：
1. 在开发机（有网）上构建好镜像
2. `docker save` 导出为 tar 文件
3. 传到内网服务器 `docker load` 导入
4. 不依赖任何外部网络
